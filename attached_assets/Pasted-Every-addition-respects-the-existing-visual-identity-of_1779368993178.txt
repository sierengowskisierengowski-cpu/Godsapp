FEATURE 1 — Onboarding Tour
Purpose
First-time users get oriented in 5 minutes without reading docs. Drops them into a real workflow.
Implementation
Interactive guided tour that fires on first launch after the authorization gate and setup wizard finish. Optional, fully skippable, persistent skip (remembered across launches).
Tour flow

Welcome card — full-screen welcome with the cloud+lightning logo pulsating, brief explainer of what GodsApp is. Two buttons: "Take the tour" / "Skip — I know what I'm doing"
Step 1 — Workspaces — highlight the workspaces sidebar item. Tooltip points to it, "Workspaces are isolated engagement environments. Each one has its own findings, evidence, notes, and scans. Let's create your first one."
Step 2 — Create workspace — opens the new-workspace dialog with sample values pre-filled and a "Use this sample workspace" button. Creates a workspace called "Tutorial — Practice Engagement" with scope 127.0.0.1, localhost, 192.168.0.0/24 pre-populated.
Step 3 — Run your first scan — guides them to Network Recon → Nmap. Pre-fills target 127.0.0.1. They click "Run Scan". The scan executes against their own machine (safe). Results appear in real-time.
Step 4 — Promote to finding — guides them to right-click on a scan result and "Promote to Finding". Sample finding gets created with auto-populated metadata, severity tag, and links to evidence.
Step 5 — Evidence locker — guides them to evidence locker. Shows them the auto-captured scan output, the chain of custody log, and the verify integrity button.
Step 6 — Notes sandbox — guides them to notes (Feature 6 below), shows them how to embed findings/evidence in notes.
Step 7 — Reports — generates a sample PDF report from the tutorial workspace. Opens it. Lets them see what their output will look like.
Step 8 — Settings tour — shows them where global settings vs per-category settings live. Walks through API keys, theme, plugins.
Step 9 — Meli integration check — if Meli is detected, shows the integration panel. If not, explains what it would unlock.
Final — "You're ready. Want to delete the tutorial workspace, or keep it for reference?" Two buttons.

Visual treatment

Each step uses the existing matrix-scramble effect on tooltip text appearance
Spotlight effect: the rest of the UI dims to ~40% opacity, the target element gets a glowing pulsating border (matches the event-border-pulse effect, but stays solid)
"Next" / "Back" / "Skip tour" controls always visible at bottom
Progress dots showing step X of 11 at top of the tooltip

Settings (Onboarding Settings sub-page in main Settings)

Re-run tour now (button)
Reset onboarding state (button — re-fires on next launch)
Tour pace (Quick / Standard / Thorough — controls how much explainer text is shown)
Enable feature spotlights on first use of new features (toggle — when a user opens a tool they've never used before, a small pulse highlights its key controls)

FEATURE 4 — Findings Deduplication & Chaining
Purpose
Multiple tools often report the same underlying issue. Auto-merge them. Also let users link related findings.
Implementation
Deduplication
When a new finding is created, GodsApp checks for similar existing findings in the same workspace based on:

Same affected asset
Same CVE reference
Same CWE classification
Same Nuclei template ID / SQLMap technique / etc.
Fuzzy match on title and description

If a match scores above a configurable threshold (default 0.85), the user is prompted:

"This finding may be a duplicate of <existing finding>. Merge?"
Options: Merge (combines evidence, keeps the higher severity), Keep both (links them as "Related"), Cancel new finding

Manual merge: select 2+ findings → right-click → Merge.
Chaining
Findings can be linked with typed relationships:

Enables — Finding A makes Finding B exploitable
Caused by — Finding A is the root cause of Finding B
Mitigates — Finding A would prevent Finding B
Related to — Generic relationship
Duplicate of — Same issue, different report

Chain visualization: in the Findings view, click "Chain View" toggle to switch from table view to a graph view showing finding relationships. Drag nodes around. Filter by severity.
Reports can include a "Finding Chains" section showing exploitation paths.
Implementation details

Dedup threshold configurable in Findings Settings
Auto-dedup on/off toggle
Always-prompt-on-creation toggle
Background dedup analysis (periodic check for duplicates in existing findings)
Chain depth limit (default 10 — prevents infinite loops)
Visual graph uses D3.js (already in tech stack per v1.0)

Settings (Findings — Dedup & Chaining sub-section)

Dedup similarity threshold slider (0.5 to 1.0)
Auto-merge if score above (slider — auto-merge for very high confidence)
Always prompt before merging (toggle)
Enabled fields for dedup matching (checkboxes: title, asset, CVE, CWE, template ID, etc.)
Chain view default layout (force-directed, hierarchical, circular)
Chain node color scheme

FEATURE 16 — Public Scope Verifier
Purpose
For bug bounty workflows: prevent accidentally scanning out-of-scope targets (which can be a CFAA violation).
Implementation
Every workspace has a scope definition. Before any tool launches a scan, GodsApp verifies the target is in-scope.
Scope definition
Workspaces have a structured scope:

In-scope IP ranges (CIDRs)
In-scope domains (with wildcard support: *.example.com)
In-scope URLs (with path patterns)
Out-of-scope items (explicit exclusions)
Authorization document (optional file attachment — engagement letter)
Authorization expiration date

Pre-scan check
When a user clicks "Run scan" on any tool, GodsApp:

Extracts the target from the scan configuration
Resolves hostname to IP if needed
Checks against in-scope and out-of-scope lists
If out-of-scope: blocks the scan, shows explanation
If in-scope but auth doc expired: warns, requires confirmation
If in-scope and auth valid: proceeds

Bypass options

Workspace setting "Allow out-of-scope" (default OFF) — user can override per workspace
One-time bypass with explicit reason logged to audit log
"Always allow this target" — adds it to in-scope after confirmation

Visual indicator
Top bar always shows current workspace's scope status:

✅ Scope valid, auth in date
⚠️ Scope valid, auth expiring soon
🚫 Scope blocking attempted out-of-scope scan

Settings (Scope Verifier sub-page)

Master enable (default ON)
Strict mode (default ON — blocks out-of-scope)
Allow override with reason (toggle)
Pre-scan check on all tools (toggle)
Warn before auth document expires (days threshold)
Log all overrides to audit log (mandatory ON)


FEATURE 17 — Snapshot & Replay
Purpose
Capture a target's state at a point in time, replay your entire engagement against it later for comparison.
Implementation
Snapshot
User clicks "Snapshot Target" on any asset:

GodsApp runs a comprehensive scan against the target (Nmap full, service detection, banner grabbing, HTTP headers if web, screenshot if web)
All output captured to evidence with a snapshot ID
Snapshot ID becomes a reference point in time

Replay
User picks a snapshot and:

Selects which past scans to replay
GodsApp re-runs each scan with same parameters against the current state of the asset
Side-by-side comparison shown
Highlights what changed: new vulns appeared, fixed vulns confirmed, configuration drift

Use case
After a pentest, the client patches. They invite you back for a re-test. You replay the entire engagement → instant report on what's fixed and what's still vulnerable.
Settings (Snapshot & Replay sub-page)

Snapshot retention (days/forever)
Auto-snapshot on engagement start (toggle)
Auto-snapshot frequency for monitored assets
Comparison sensitivity (what counts as a change)


FEATURE 18 — Safe Mode
Purpose
Limit dangerous tools for training, students, supervised use.
Implementation
Toggle in Settings → General → Safe Mode.
When Safe Mode is ON:

Active exploitation tools are disabled
Mass scanners (Masscan with rate >100, Nmap with -T5) are throttled
Destructive options (e.g. SQLMap with --batch and --random-agent at the same time) require additional confirmation
Reverse shell generation requires explicit acknowledgment
Metasploit bridge disabled
Payload generation disabled
All other tools work normally

Safe Mode can be set per workspace (e.g. enable for "Tutorial — Practice Engagement" workspace, disable for production engagements).
Padlock indicator
Sidebar shows a small padlock next to disabled tools when Safe Mode is on. Tooltip explains why.
Settings (Safe Mode sub-page)

Master toggle
Per-tool override (advanced users can mark specific tools as always-allowed)
Per-workspace override
Require admin password to disable (for parental control / supervised use)
Audit log on every Safe Mode toggle
